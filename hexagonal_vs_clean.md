# Hexagonal Architecture vs Clean Architecture

## Hexagonal Architecture (Ports and Adapters)

### Определение

**Hexagonal Architecture** («Гексагональная архитектура») — это паттерн проектирования, целью которого является разделение бизнес-логики приложения от внешних интерфейсов (UI, БД, API и т.д.) с помощью абстракций — **портов** и **адаптеров**.

Ядро приложения (доменная модель) находится в центре гексагона и не зависит от внешнего мира. Внешние взаимодействия реализуются через интерфейсы (порты), которые реализуются адаптерами — с одной стороны (вход), с другой (выход).

---

## Clean Architecture

### Оределение

**Clean Architecture** — это архитектурный подход, предложенный Робертом Мартином (Uncle Bob), направленный на создание **независимых**, **тестируемых** и **удобных для поддержки** систем. Основная идея — **направленная зависимость внутрь**: внешние слои зависят от внутренних, но не наоборот.

Ядро (entities, use cases) должно быть изолировано от фреймворков, инструментов и интерфейсов. Каждый слой отвечает за свою область ответственности и не должен «знать» о реализации внешних деталей.

---

## Сравнение: Hexagonal vs Clean Architecture

| Критерий | Hexagonal Architecture | Clean Architecture |
|----------|------------------------|---------------------|
| **Цель** | Разделить доменную логику от инфраструктуры через абстракции (порты/адаптеры) | Делить систему на слои с направленной зависимостью внутрь |
| **Ключевая идея** | Домен — в центре гексагона; взаимодействие — через порты | Зависимости направлены внутрь: внешние слои зависят от внутренних |
| **Структура** | Центр — домен. Два вида портов: входные (input) и выходные (output). Адаптеры — по краям. | Четыре (или пять) concentric слоя: Entities → Use Cases → Controllers/Interface Adapters → Frameworks/Drivers |
| **Зависимости** | Домен не зависит от внешних слоёв. Порты — интерфейсы, реализуемые адаптерами. | **Правило зависимости**: внешние слои могут зависеть от внутренних, но не наоборот. |
| **Тестирование** | Очень удобно: можно заменить адаптеры (например, `MemoryRepo`) в тестах без изменения ядра. | Также легко, но тесты могут зависеть от слоя Use Cases и их интерфейсов. |
| **Гибкость внедрения** | Высокая: новые входы/выходы — новые адаптеры. Нет ограничения на количество. | Гибкая, но структура более жёсткая: строгая иерархия слоёв. |
| **Абстракции** | Порты (интерфейсы) — *для входа и выхода*. | Интерфейсы — на границе слоёв, но не все интерфейсы выделяются как «порты». |
| **Фокус на роли** | Адаптеры — «переводчики» между внешним миром и ядром. | Слои — роли в системе: сущности, бизнес-правила, адаптеры интерфейсов, инструменты. |
| **Название** | От геометрии: домен — шестиугольник, порты — вершины. | От «чистоты»: домен не загрязнён инфраструктурой. |

## Сравнение слоёв/ролей (в терминах страхового приложения)

| Hexagonal Architecture | Clean Architecture | Роль | Примечание |
|------------------------|---------------------|------|------------|
| **Domain / Core** | **Entities** | Бизнес-сущности (например, `Policy`, `Claim`) | Простые объекты с бизнес-правилами. Без поведения или с минимальным. |
| | **Use Cases** | Бизнес-операции и сценарии (например, `CalculatePremiumUseCase`) | Содержит бизнес-логику; взаимодействует с портами. |
| **Input Port** | | Интерфейс для входных действий (например, `ICalculatePremium`) | Указывает *что* делать, но не *как*. |
| **Input Adapter** | **Controllers / Presenters** | Приводит внешний запрос к интерфейсу Use Cases (REST, CLI) | Преобразует внешний формат (JSON) в объекты ядра. |
| **Output Port** | **Interface Adapters** | Интерфейс для выходных действий (например, `IPolicyRepository`) | Граница, где домен «отдаёт» данные. |
| **Output Adapter** | **Frameworks & Drivers** | Реализация внешнего мира (БД, API, email, файлы) | Технологический слой: может зависеть от домена, но не наоборот. |

> 💡 В Clean Architecture *Interface Adapters* — это слой, который может содержать DTO, мappers, контроллеры (в терминах MVC). В Hexagonal они распределены по адаптерам (входные и выходные).

## Визуальное сравнение

### Hexagonal Architecture

```
         +------------------+
         |  Input Adapter   |   (REST, CLI, GUI)
         +------------------+
                   ▲│
         ┌─────────┼│───────────┐
         │         ▼│           │
   +-------------------+     +-------------------+
   |  Input Port       │     |  Output Port      |
   |  (e.g., IUseCase) |     | (e.g., IRepository) |
   +-------------------+     +-------------------+
         │         ▲│           │
         │         └────────────┘
   +-----------------------------+
   |        Core Domain          |
   |   (Entities, Business Rules)|
   +-----------------------------+
         ▲         ┌────────────┐
         │         ▼            │
   +-------------------+     +-------------------+
   |  Output Adapter   |     |  Input Adapter    |
   | (DB, Email, etc.) |     | (REST, CLI, GUI)  |
   +-------------------+     +-------------------+
```

### Clean Architecture

```
+--------------------------------+
|    Frameworks & Drivers        | ← Web, DB, UI frameworks
+--------------------------------+
|     Interface Adapters         | ← Controllers, Presenters, DTOs
+--------------------------------+
|         Use Cases              | ← Business logic orchestration
+--------------------------------+
|         Entities               | ← Core domain models
+--------------------------------+
```

## Какой выбрать?

| Сценарий | Рекомендация |
|----------|--------------|
| Необходимо **быстро менять входы и выходы** (REST → GraphQL → CLI) | ✅ Hexagonal — явное разделение портов и адаптеров. |
| Приложение **сложное и многослойное**, важна строгая изоляция бизнес-правил | ✅ Clean — слоистость делает структуру более предсказуемой. |
| Требуется **максимальная гибкость** и **простота тестирования** | ✅ Hexagonal — каждый адаптер можно тестировать отдельно. |
| Команда уже работает с **чистыми архитектурами**, понимает паттерн слоёв | ✅ Clean — удобно для команд,熟悉 Onion/Clean. |
| Проект **небольшой**, бизнес-логика не сложная | ⚠️ Оба — могут быть избыточны. |

## Заключение

- **Hexagonal Architecture** делает акцент на **взаимодействии через абстракции** — домен не знает, откуда приходят данные и куда они уходят.
- **Clean Architecture** делает акцент на **направлении зависимостей** — ядро чистое, всё остальное зависит от него, но не наоборот.

Оба подхода стремятся к одной цели: **создать гибкую, тестируемую и устойчивую к изменениям систему**, где бизнес-логика не ломается от смены UI или БД.

Они часто комбинируются: например, в Clean Architecture используются порты и адаптеры внутри слоёв интерфейсов и фреймворков.
